## はじめに

JavaScriptからRustに入門した時、こんなコードを見て混乱しました。

```rust
std::env::var("API_KEY")
```

「`std` の中の `env` の中の `var` を呼んでる？`env` ってクラスなの？」

結論から言うと、**`env` はクラスではなくモジュールです**。そしてクラスとモジュールは全く別物です。

---

## モジュールとは

**コードを整理するための名前空間（フォルダのようなもの）**です。

```rust
// モジュール = 関数や型をまとめる箱
std::env::var("API_KEY");
// std   = 標準ライブラリ（クレート）
// env   = 環境変数関連をまとめたモジュール
// var   = 関数
```

JavaScriptで言うと**ファイルのパス**に近いイメージです。

```javascript
// JavaScript（イメージ。var は JS の予約語なので実際にはこの名前では import できません）
import { getVar } from 'std/env'
```

モジュールは**インスタンスを作らない**のが特徴です。直接関数を呼ぶだけです。

---

## クラスとは

**データと処理をひとまとめにするもの**です。

```javascript
// JavaScript
class TwitterClient {
    constructor(apiKey) {
        this.apiKey = apiKey  // データ
    }

    postTweet(content) {      // 処理
        // ...
    }
}

// インスタンスを作って使う
const client = new TwitterClient("abc123")
client.postTweet("テストツイートです")
```

クラスは**インスタンスを作る**のが特徴です。

---

## Rustにクラスはない

Rustにはクラスという概念がありません。代わりに **struct（構造体）+ impl** を使います。

```rust
// Rustでのクラス相当
struct TwitterClient {
    api_key: String,   // データ
}

impl TwitterClient {
    // コンストラクタ相当
    fn new(api_key: String) -> Self {
        Self { api_key }
    }

    // メソッド相当
    fn post_tweet(&self, content: &str) {
        // ...
    }
}

// インスタンスを作って使う
let client = TwitterClient::new("abc123".to_string());
client.post_tweet("テストツイートです");
```

---

## 本質：存在のレイヤーが違う

ここまで読んで「結局、構文が違うだけでは？」と感じるかもしれません。しかしモジュールとクラスは、**そもそもプログラムの中で「いつ・どう存在するか」が違います**。

|  | モジュール | クラス（struct + impl） |
|---|---|---|
| 存在する時点 | **コンパイル時に解決される**（静的） | **実行時に値として存在する**（動的） |
| 実体 | 名前空間。パスを解決するルール | 型と、その型のインスタンスを作る仕組み |
| 個数 | プログラム中に1つだけ | インスタンスを何個でも作れる |
| 状態 | 持たない（中の関数や定数を参照するだけ） | インスタンスごとに状態を持つ |

たとえば `std::env::var("API_KEY")` は、コンパイラが `std → env → var` というパスをたどって**関数そのもの**を見つけて呼びます。`env` という値はどこにも存在しません。

一方 `let client = TwitterClient::new(...)` で作った `client` は、**実行時にメモリ上に存在するデータ**です。`api_key` という状態を抱えていて、同じ型からいくつでも別のインスタンスを作れます。

### 呼び出し記号 `::` と `.` がこの違いを表している

Rust ではこの本質の違いが、**呼び出し記号の使い分け**として構文に現れています。

```rust
std::env::var("API_KEY")     // :: → 名前空間をたどる（静的なパス解決）
client.post_tweet("hello")   // .  → インスタンスにメッセージを送る（動的な呼び出し）
```

- `::` … コンパイル時にパスを解決する。レシーバー（呼び出し対象のインスタンス）はいない
- `.` … 実行時のインスタンスに対して呼ぶ。左辺には必ず値がある

JavaScript ではどちらも `.` で書けてしまうため、`Module.func()` と `instance.method()` が見た目上区別できませんでした。Rust は記号レベルで強制的に分けることで、「いま自分は静的なパスをたどっているのか、それとも実行時のインスタンスを操作しているのか」を常にコードから読み取れるようにしています。

---

## まとめ対応表

| JavaScript | Rust | 役割 |
|---|---|---|
| `class` | `struct` + `impl` | データと処理をまとめる |
| `import` / フォルダ構造 | `mod` / モジュール | コードを整理する |
| `new ClassName()` | `ClassName::new()` | インスタンスを作る |
| `this.method()` | `self.method()` | 自身のメソッドを呼ぶ |

---

## おわりに

混乱のポイントを一言でまとめると：

- **モジュール** → コードを整理するフォルダ。インスタンスは作らない
- **クラス相当** → `struct` + `impl` の組み合わせ。インスタンスを作る

`std::env::var` を見て「`env` ってクラス？」と思ったら、「ただの整理棚だ」と思い出してください。
