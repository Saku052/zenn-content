## はじめに
最近、個人で開発しているRustのアプリケーションで、エラーハンドリングについて考えるようになりました。
具体的には...

``` rust
.expect("API_KEYが設定されていません")
```

vs

``` rust
.unwarp()
```

vs

``` rust
.context("API_KEYが設定されていません")
```

キッカケは `.unwrap()`で書いていたらclaudeにやめなさいと言われた事です。
どちらもエラーが起こった際に　`("API_KEYが設定されていません")`　というメッセージを表示してくれるのですが、その細かい挙動が少しだけ違います。
今回はその挙動について少し調べてみたので、まとめていきます。

## expectとは：ありえない状態になった場合に強制終了
#### 主な使い方
・開発途中でまだ具体的なエラー内容が決まっていない場合
・コード的にありえない状態で、万が一起こってしまった場合データ破損につながるケース等

簡単に言えば別にエラーメッセージを出すだけならexpectでOKな気がします。ただ、厳密にこういうエラーが起こった場合はこれを返すみたいなのが決まってるケースだと最後の方法の方が適切だと思います。

## unwrap : OptionやResult型から値を取り出す
####　主な使い方
・OptionやResult型から値を取り出したい場合
・まだ開発途中でエラーの内容が決まっていない場合

結論から言うと、unwrapに関しては業界的にあまりおすすめされていない様に思えました。ただ、ユースケースとしてはまだ開発途中でエラーの内容はまだわからないけど、将来的にcontextに移行したいと考えている場合などはあるかなと思います。

## context：OptionやResult型から値を取り出し、Errに何が起こったのかメッセージを残す
####　主な使い方
・エラーが起こった場合何して欲しいのか理解しているケース。
・一応本番環境でも想定しうるエラー

.contextにする場合は返り値として、Resultを返します。Okの時はそのまま返しますが、エラーが起こった場合はErrと一緒にcontextの中身を返す形になります。
本番環境にデプロイする前はこちらのcontextにするのが業界的なベストプラクティスだと思います。

## 3つの使い分けまとめ
結論個人的には、context使っておけばいいのかなと思います。ただ、contextを使う場合はエラーが起こったときにどう言う処理を行うかの定義も行わないといけなくて、書かないといけないコードが増量すると言うデメリットもあるので...
・個人開発
・小規模開発
ではexpect使っちゃってもいいのかなと思います。

unwrapは使わなくていいと思います。
